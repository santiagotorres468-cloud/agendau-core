👤 Casos de Uso — Actor: Estudiante
CU-03
Visualización del Calendario Público de Oferta
👤 Estudiante / Usuario Anónimo 🌐 Acceso Público
📋 Descripción
Despliegue de un calendario interactivo en la interfaz principal que muestra la oferta académica disponible, incluyendo fecha, hora y cupos de cada tutoría.
✅ Precondiciones
Ninguna — acceso completamente público sin autenticación.
🔄 Flujo Principal
  • 1
    El usuario ingresa a la URL principal de la plataforma AgendaU.
  • 2
    El sistema consulta la base de datos y renderiza un calendario mensual/semanal interactivo.
  • 3
    El sistema despliega bloques visuales con fecha, hora y cupos disponibles de cada tutoría ofertada.
CU-04
Consulta de Detalles de Ubicación y Modalidad
👤 Estudiante 🔗 Incluido en CU-03
📋 Descripción
Permite a los estudiantes visualizar la información logística específica de una tutoría (ubicación física o enlace virtual) antes de realizar una reserva.
✅ Precondiciones
El calendario público debe estar visible y el estudiante debe haber seleccionado una tutoría (CU-03).
🔄 Flujo Principal
  • 1
    El estudiante hace clic sobre una tutoría específica en el calendario.
  • 2
    El sistema despliega una ventana modal con los detalles estructurados de la sesión.
  • 3
    El sistema especifica claramente la Modalidad (Presencial/Virtual) y la Ubicación exacta: Sede, Bloque y Aula, o Enlace de conexión según corresponda.
CU-06
Reserva de Cupo en Tutoría con Validación de Identidad
👤 Estudiante ⚡ Extiende CU-03
📋 Descripción
Proceso mediante el cual un estudiante asegura un espacio en una tutoría, validando su documento de identidad contra los registros institucionales activos.
✅ Precondiciones
Debe existir al menos un (1) cupo disponible en la tutoría seleccionada.
🔄 Flujo Principal
  • 1
    El estudiante selecciona "Reservar Cupo" en los detalles de una tutoría.
  • 2
    El sistema despliega un formulario solicitando el número de identificación (Cédula / T.I.).
  • 3
    El estudiante ingresa su documento y confirma la solicitud.
  • 4
    El sistema cruza el documento con la base de datos de estudiantes activos de la institución.
  • 5
    El sistema registra la reserva, descuenta un cupo del aforo total y emite una confirmación en pantalla.
⚠ Flujo Alternativo (4a) — Documento no encontrado
Si el documento ingresado no existe en los registros institucionales, el sistema bloquea la reserva y emite una alerta indicando que el estudiante no está activo en el sistema.
CU-07
Cancelación de Reserva de Tutoría
👤 Estudiante 🔗 Desencadena CU-08
📋 Descripción
Permite a un estudiante anular una reserva previamente efectuada, renunciando a su cupo y liberando el espacio para otro participante.
✅ Precondiciones
El estudiante debe poseer una reserva activa registrada en el sistema.
🔄 Flujo Principal
  • 1
    El estudiante accede a la opción de consulta/cancelación de reservas desde su panel.
  • 2
    El estudiante selecciona la tutoría que desea cancelar de su listado de reservas activas.
  • 3
    El sistema solicita una confirmación explícita de la acción.
  • 4
    El estudiante confirma la anulación.
  • 5
    El sistema elimina lógicamente el registro de la base de datos y notifica el éxito de la operación.
👤 Casos de Uso — Actor: Docente
CU-09
Visualización de Agenda Personalizada del Docente
👤 Docente 🔐 Requiere CU-05
📋 Descripción
Permite al personal docente consultar un cronograma exclusivo con las sesiones de tutoría que debe impartir, filtrado únicamente por su perfil de usuario.
✅ Precondiciones
El docente debe estar autenticado en el sistema (CU-05) con rol "Docente".
🔄 Flujo Principal
  • 1
    El docente ingresa a su panel de control "Centro de Control".
  • 2
    El sistema filtra y renderiza únicamente las tutorías asociadas a su perfil de usuario.
  • 3
    El docente visualiza el listado detallando fecha, hora, modalidad y ubicación física/virtual asignada.
CU-10
Consulta del Listado de Estudiantes Inscritos
👤 Docente 🔗 Incluido en CU-11
📋 Descripción
Permite al docente conocer anticipadamente la cantidad e identidad de los estudiantes que asistirán a su sesión de tutoría.
✅ Precondiciones
El docente debe estar autenticado y tener al menos una tutoría programada con estudiantes inscritos.
🔄 Flujo Principal
  • 1
    El docente selecciona una tutoría específica desde su agenda personal (CU-09).
  • 2
    El sistema ejecuta una consulta en la base de datos de reservas activas.
  • 3
    El sistema despliega una tabla con los nombres y documentos de identidad de los estudiantes inscritos.
CU-11
Registro de Control de Asistencia a Sesión
👤 Docente ⚡ Habilita CU-12, CU-13
📋 Descripción
Módulo para formalizar la participación efectiva de los estudiantes que reservaron cupo, diferenciando entre asistentes y ausentes.
✅ Precondiciones
La tutoría debe estar en ejecución o haber finalizado. El listado de inscritos debe ser visible (CU-10).
🔄 Flujo Principal
  • 1
    El docente visualiza el listado de inscritos (CU-10).
  • 2
    El docente selecciona la opción "Registrar Asistencia".
  • 3
    El sistema habilita controles de selección por alumno: Asistió / No Asistió.
  • 4
    El docente marca el estado correspondiente para cada alumno y presiona "Guardar".
  • 5
    El sistema actualiza el estado de asistencia en la base de datos y confirma la operación.
CU-12
Registro de Observaciones de Evolución Académica
👤 Docente ⚡ Extiende CU-11
📋 Descripción
Permite la captura de datos cualitativos respecto al desempeño del estudiante durante la asesoría: fortalezas, debilidades y hallazgos académicos.
✅ Precondiciones
El estudiante debió ser marcado como "Asistió" en el control de asistencia (CU-11).
🔄 Flujo Principal
  • 1
    El docente selecciona "Agregar Evolución" frente al nombre de un estudiante asistente.
  • 2
    El sistema despliega un área de texto para redacción libre.
  • 3
    El docente redacta los hallazgos, fortalezas o debilidades evidenciadas durante la sesión.
  • 4
    El sistema guarda la observación, asociándola al registro histórico del estudiante.
CU-13
Generación de Informe Consolidado de Seguimiento
👤 Docente 🔗 Requiere CU-11 y CU-12
📋 Descripción
Funcionalidad para exportar un soporte documental formal de la sesión impartida, consolidando datos de asistencia y observaciones académicas en formato PDF.
✅ Precondiciones
Debe existir registro de asistencia (CU-11) y observaciones de evolución guardadas (CU-12).
🔄 Flujo Principal
  • 1
    El docente hace clic en "Generar Informe PDF" en el perfil de un estudiante.
  • 2
    El sistema recopila datos de la tutoría, datos del estudiante y las observaciones ingresadas.
  • 3
    El sistema formatea la información y renderiza un documento PDF estructurado.
  • 4
    El sistema inicia la descarga automática del archivo en el equipo del docente.
CU-14
Gestión de Privacidad de Registros Académicos
👤 Docente ⚙️ Sistema / Seguridad
📋 Descripción
Regla de negocio automatizada que garantiza la confidencialidad de las observaciones académicas, restringiendo el acceso únicamente al docente autor del registro.
✅ Precondiciones
Existencia de registros de evolución académica creados (CU-12).
🔄 Flujo Principal
  • 1
    Un usuario intenta acceder a la evolución académica de un estudiante.
  • 2
    El sistema valida el ID del usuario solicitante contra el ID del docente creador de la tutoría.
  • 3
    Si coinciden (es el docente autor), el sistema permite la visualización y exportación del informe.
⚠ Flujo Alternativo (3a) — Acceso no autorizado
Si el ID no coincide (otro docente o administrador), el sistema oculta los campos de texto y bloquea la acción, protegiendo la privacidad del historial cualitativo del estudiante.
👤 Casos de Uso — Actor: Administrador
CU-05
Autenticación y Control de Acceso (RBAC)
👤 Administrador / Docente 🔐 Compartido
📋 Descripción
Proceso de inicio de sesión seguro que asigna permisos y vistas diferenciadas basadas en el rol del usuario (Administrador o Docente), restringiendo rutas no autorizadas.
✅ Precondiciones
El usuario debe poseer credenciales válidas creadas previamente por el administrador del sistema.
🔄 Flujo Principal
  • 1
    El usuario accede a la vista de "Login" del portal administrativo.
  • 2
    El usuario ingresa su correo electrónico institucional y contraseña.
  • 3
    El sistema encripta y valida las credenciales contra la base de datos de usuarios.
  • 4
    El sistema identifica el rol asignado y redirige al panel de control correspondiente, bloqueando rutas no autorizadas.
⚠ Flujo Alternativo (3a) — Credenciales incorrectas
Si las credenciales son incorrectas o el usuario está inactivo, el sistema deniega el acceso y muestra una alerta de error sin revelar información sensible.
CU-01
Carga Masiva de Horarios de Tutorías
👤 Administrador 🔗 Desencadena CU-02
📋 Descripción
Permite al administrador importar un archivo Excel estandarizado para registrar múltiples horarios de tutorías de forma simultánea, reduciendo la carga operativa manual.
✅ Precondiciones
El administrador debe estar autenticado en el sistema (CU-05) con rol "Administrador".
🔄 Flujo Principal
  • 1
    El administrador accede al módulo "Gestión de Horarios" → "Carga masiva".
  • 2
    El administrador adjunta el archivo Excel con los campos requeridos: Materia, Docente, Correo, Modalidad, Sede, Bloque, Aula.
  • 3
    El sistema procesa el archivo, ejecuta la validación (CU-02) y almacena los registros en la base de datos.
  • 4
    El sistema notifica el éxito de la operación mostrando la cantidad de horarios cargados y las filas omitidas con su motivo.
CU-15
Ejecución de Respaldo de Seguridad (Backup)
👤 Administrador del Sistema 🛡️ Contingencia
📋 Descripción
Procedimiento de contingencia para preservar el código fuente y los datos de la plataforma mediante volcado completo de la base de datos relacional.
✅ Precondiciones
El administrador debe poseer privilegios máximos en el entorno de despliegue del servidor.
🔄 Flujo Principal
  • 1
    El administrador accede al panel de herramientas del servidor/sistema.
  • 2
    El administrador ejecuta el comando o botón "Generar Copia de Seguridad".
  • 3
    El sistema realiza un dump (volcado) completo de la base de datos relacional.
  • 4
    El sistema almacena el archivo comprimido y notifica la finalización exitosa del respaldo.
⚙️ Casos de Uso — Actor: Sistema (Automatizado)
CU-02
Validación de Integridad de Plantilla de Horarios
⚙️ Sistema / Administrador 🔗 Incluido en CU-01
📋 Descripción
Mecanismo de validación automática que garantiza que los archivos Excel cargados cumplan con la estructura de datos obligatoria antes de su inserción en la base de datos.
✅ Precondiciones
El administrador debe haber iniciado la carga de un archivo Excel (CU-01).
🔄 Flujo Principal
  • 1
    El sistema intercepta el archivo Excel antes de su inserción en la base de datos.
  • 2
    El sistema verifica que los campos obligatorios estén presentes y con formato correcto según la modalidad.
  • 3
    El sistema aprueba la estructura y continúa con el proceso de carga (CU-01).
⚠ Flujo Alternativo (2a) — Estructura inválida
Si el archivo presenta celdas vacías, formatos incorrectos o columnas faltantes, el sistema aborta la operación y despliega un reporte detallado con las filas que contienen errores para su corrección.
CU-08
Actualización Dinámica de Disponibilidad de Cupos
⚙️ Sistema Automatizado 🔗 Disparado por CU-07
📋 Descripción
Proceso automatizado en segundo plano que recalcula el aforo disponible tras cada cancelación, manteniendo el calendario público actualizado en tiempo real.
✅ Precondiciones
Debe haberse ejecutado exitosamente una cancelación de reserva (CU-07).
🔄 Flujo Principal
  • 1
    El sistema detecta automáticamente la eliminación lógica de una reserva en la base de datos.
  • 2
    El sistema incrementa en +1 el contador de cupos disponibles para la sesión afectada.
  • 3
    El sistema refresca el estado en el calendario público, permitiendo que otro estudiante ocupe la vacante liberada.
